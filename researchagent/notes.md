16) **GPT Architecture Deep Dive** — **CausalSelfAttention:** Each token looks at previous tokens only (not future) to decide what comes next. Q (Query): "What am I looking for?", K (Key): "What do I contain?", V (Value): "Here's my actual info". Q asks → K matches → V provides content. `__init__`: Sets up tools (weights/layers) once — qkv_proj, out_proj, mask. `forward`: Uses those tools on every batch. Output projection: Blends all attention heads into one unified output (learned mixing). Mask: Lower triangular matrix that blocks future tokens (1 = see, 0 = blocked). Enforces "no peeking ahead". Softmax: Converts raw scores to probabilities (0-1, sum to 1). Dropout: Randomly zeroes attention weights during training (regularization). Regularization: Prevents memorizing training data — forces model to generalize, not memorize. **FeedForward:** Simple MLP that processes each token independently after attention. Structure: Linear (expand 4x) → GELU (filter what matters) → Linear (shrink back) → Dropout. Attention = tokens talk to each other. FeedForward = each token digests info alone. The weights are learned during training. **TransformerBlock:** One complete layer: LayerNorm → Attention → Add → LayerNorm → FeedForward → Add. Residual connection (+): Adds original input back, preserves info. LayerNorm: Stabilizes values. GPT stacks 6 blocks — each round builds deeper understanding. What each round learns is automatic (emerges from training), not manually defined. **GPT Class:** Complete model: Token Embedding + Position Embedding → 6 TransformerBlocks → LayerNorm → Linear head. `forward`: Run input through model, return prediction/loss. `generate`: Autoregressive text generation (predict → append → repeat).

17) **Step 8: train.py — Data Loading (DataLoader class)** — Feeds training data to the model in batches. **How it works:** (1) Reads parquet files, (2) Tokenizes text → token IDs, (3) Fills a buffer with tokens, (4) Creates batches of (input, target) pairs. **Input/Target relationship:** Input = [Once, upon, a, time, the], Target = [upon, a, time, the, princess] (shifted by 1 — predict next token). **Key methods:** `_load_next_file()` loads parquet file into memory, `_get_next_tokens()` tokenizes one document, `_fill_buffer()` keeps buffer full of tokens, `get_batch()` returns (inputs, targets) tensors ready for GPU. **Split:** Training uses all files except last, validation uses last file only.

18) **Step 9: train.py — Training Loop** — The evaluate() function runs the model on the validation set with dropout OFF (model.eval()) and returns val_bpb (bits per byte) — a vocab-size independent metric where lower is better, allowing fair comparison across different experiments. ## Summary:
This project is an autonomous AI research agent designed to iteratively improve LLM training code without human supervision. An AI agent (Claude or Codex) automatically modifies train.py, executes a short 5-minute training experiment, and evaluates whether the validation loss improved using val_bpb (validation bits per byte) as the metric — lower is better. If the change yields better results, it's kept; otherwise discarded. This cycle repeats continuously overnight, potentially running approximately 100 experiments. The fork adds macOS Apple Silicon (MPS) support to Andrej Karpathy's original autoresearch project.

The system has a minimal 3-file architecture: prepare.py handles one-time data download and tokenizer training, train.py contains the GPT model and training loop that the AI agent modifies, and program.md provides instructions for the AI agent. The experiment loop follows this pattern: AI reads program.md → modifies train.py → runs 5-minute training → checks if val_bpb improved → keeps or discards changes → repeat. train.py is the critical file that the AI agent actively experiments with, containing hyperparameters that it modifies to find optimal configurations.

prepare.py establishes key constants: MAX_SEQ_LEN=2048 (context window), TIME_BUDGET=300s (5 minutes per experiment), EVAL_TOKENS=~21M (validation tokens), VOCAB_SIZE=8192 (small vocab for faster training), and CACHE_DIR=~/.cache/autoresearch (data and tokenizer storage). Data download uses download_single_shard() with 5 retries and exponential backoff (2s, 4s, 8s, 16s) and download_data() with parallel multiprocessing Pool. Key reliability features include atomic writes via temp file → rename, skipping already-downloaded files, and exponential backoff on failures.

Understanding the distinction between tokenizer training and model training is critical. These are two separate processes: tokenizer training (prepare.py) learns how to split text into tokens, producing a dictionary/merge table that maps text like "hello" to tokens [hel, lo], taking approximately 1 minute to complete once and remaining unchanged by the AI. Model training (train.py), by contrast, learns to predict the next token using neural network weights, learning patterns like "Once upon a" → "time", taking 5 minutes per experiment across 100+ experiments, and being actively modified by the AI agent. The analogy: tokenizer training creates an alphabet or dictionary, while model training learns to write stories using that alphabet.

The train.py file contains both the hyperparameters and the model architecture that the AI agent experiments with during the optimization process. Hyperparameters include architectural settings: N_LAYERS=6 (Transformer depth/number of layers), N_HEADS=6 (number of attention heads), N_EMBED=384 (embedding dimension size), and DROPOUT=0.1 (regularization to prevent overfitting). Training hyperparameters include BATCH_SIZE=32 (sequences processed per batch), LEARNING_RATE=3e-4 (Adam optimizer learning rate), and WEIGHT_DECAY=0.1 (L2 regularization strength). These values are standard starting points from GPT research, chosen as baselines rather than optimal values — the AI agent experiments with variations overnight to discover better configurations. The file also implements intelligent device detection with priority order: CUDA (NVIDIA GPUs) > MPS (Apple Silicon) > CPU, ensuring optimal hardware utilization across different platforms.

The GPT model architecture in train.py consists of four main classes implementing the Transformer architecture. CausalSelfAttention implements multi-head attention where each token can only attend to previous tokens (not future ones), using Query, Key, Value projections plus a causal mask to enforce left-to-right information flow. FeedForward is an MLP (multi-layer perceptron) that runs after attention, consisting of Linear → GELU → Linear layers, expanding to 4x the embedding dimension then projecting back. TransformerBlock combines one attention layer and one feedforward layer with residual connections and LayerNorm in pre-norm style (LayerNorm before attention/FFN, which is more stable). The GPT class ties everything together: token embeddings + position embeddings feed into N stacked TransformerBlocks, followed by LayerNorm and a Linear layer to predict the next token. It uses weight tying between input and output embeddings for efficiency and includes a .generate() method for autoregressive text generation. Key architectural concepts include the causal mask preventing future token visibility, residual connections (x + layer(x)) for stable gradient flow during training, and pre-norm architecture for improved training stability.

The attention mechanism uses three learned transformations: Q (Query) asks "What am I looking for?", K (Key) answers "What do I contain?", and V (Value) provides "Here's my actual info". The flow is: Q asks → K matches → V provides content. Step 8 introduces the DataLoader class, which handles the critical task of feeding training data to the model in efficient batches. The DataLoader follows a four-step process: reading parquet files from disk, tokenizing text into token IDs using the trained tokenizer, filling an internal buffer with tokens for efficient access, and creating batches of (input, target) pairs ready for GPU processing. The input/target relationship is fundamental to language modeling: inputs and targets are identical sequences offset by one token position. For example, if the input sequence is [Once, upon, a, time, the], the target sequence is [upon, a, time, the, princess] — shifted by exactly one position so the model learns to predict the next token. The DataLoader implements several key methods: `_load_next_file()` loads parquet files into memory, `_get_next_tokens()` tokenizes one document at a time, `_fill_buffer()` maintains a full token buffer for smooth batch generation, and `get_batch()` returns properly formatted (inputs, targets) tensors ready for GPU consumption. The data split strategy is straightforward: training uses all parquet files except the last one, while validation uses only the last file, ensuring clean separation between training and validation data without overlap.

Step 9 completes train.py with the training loop implementation. The evaluate() function runs the model on the validation set with dropout disabled (via model.eval()) and returns val_bpb (validation bits per byte) — a vocabulary-size independent metric where lower values indicate better performance, enabling fair comparison across different experiments with different hyperparameters. The train() function implements the main training loop with a clear five-step process: load the tokenizer from disk, create the GPT model instance with current hyperparameters, create the AdamW optimizer, execute the training loop for TIME_BUDGET (5 minutes) where each iteration gets a batch of (inputs, targets), performs a forward pass where loss = model(inputs, targets) calculates prediction error, performs a backward pass where loss.backward() computes gradients via backpropagation, and updates weights via optimizer.step(), and finally evaluate the trained model and report val_bpb. The training loop implements critical concepts: optimizer.zero_grad() clears old gradients before each step to prevent accumulation from previous batches, loss.backward() performs backpropagation to compute how each weight should be adjusted, optimizer.step() actually updates the model weights based on the computed gradients, and model.train() / model.eval() toggle dropout behavior on or off for training versus evaluation. The entry point `if __name__ == "__main__": train()` makes train.py directly executable as a standalone script, allowing both the AI agent and human developers to run training experiments simply by executing the file.

Step 10 introduces program.md, the instruction file that tells the AI agent (Claude or Codex) what actions to perform during overnight experiments. The AI agent executes a continuous loop consisting of five steps: modify train.py by adjusting hyperparameters or architecture, run `python train.py` for a fixed 5-minute training session, check the resulting val_bpb metric, keep the modification if val_bpb improved (lower is better) or discard it if val_bpb worsened, and repeat. The AI agent has permission to modify specific aspects of train.py: hyperparameters including N_LAYERS, N_HEADS, N_EMBED, DROPOUT, BATCH_SIZE, LEARNING_RATE, and WEIGHT_DECAY; architecture choices like activation functions, position embeddings, and attention patterns; and training configurations such as optimizers, learning rate schedulers, gradient clipping, and mixed precision settings.

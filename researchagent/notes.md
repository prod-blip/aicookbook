## Raw Notes:
1) This project is an autonomous AI research agent designed to iteratively improve LLM training code without human supervision. An AI agent (Claude or Codex) automatically modifies train.py, executes a short 5-minute training experiment, and evaluates whether the validation loss improved using val_bpb (validation bits per byte) as the metric — lower is better. If the change yields better results, it's kept; otherwise discarded. This cycle repeats continuously overnight, potentially running ~100 experiments. The fork adds macOS Apple Silicon (MPS) support to Andrej Karpathy's original autoresearch project.

2) The system has a minimal 3-file architecture: prepare.py (one-time data download + tokenizer training), train.py (GPT model + training loop that the AI agent modifies), and program.md (instructions for the AI agent). The experiment loop: AI reads program.md → modifies train.py → runs 5-min training → checks if val_bpb improved → keeps or discards → repeat.

3) prepare.py establishes key constants: MAX_SEQ_LEN=2048 (context window), TIME_BUDGET=300s (5 min per experiment), EVAL_TOKENS=~21M (validation tokens), VOCAB_SIZE=8192 (small vocab for faster training), CACHE_DIR=~/.cache/autoresearch (data + tokenizer storage). Data download uses download_single_shard() (5 retries + exponential backoff: 2s, 4s, 8s, 16s) and download_data() (parallel multiprocessing Pool). Key reliability features: atomic writes via temp file → rename, skip already-downloaded files, exponential backoff on failures.

Four components: list_parquet_files(), text_iterator() (stops at 100M chars), train_tokenizer() (trains + saves to disk), and Tokenizer class (encode/decode). Only needs ~50M characters to learn good merges, trained once, reused for all experiments. Uses tiktoken GPT-2 base for simplicity.

5) The TinyStories dataset (karpathy/tinystories-gpt4-clean) was chosen for initial experimentation: ~673MB total, 2.7M stories, clean GPT-4 generated data, simple vocabulary, easy to test via story continuations.

6) Shards = dataset split into multiple parquet files for easier downloading/processing. Downloaded once at setup (prepare.py), then reused for all experiments — the experiment loop does NOT re-download data. TinyStories has 2-10 shards (~50-100MB each). For experimentation, 2 shards (~200MB) is enough to train a small GPT model that generates coherent children's stories and proves the autoresearch loop works, running entirely on Mac. TinyStories tokenizer learns "princess", "bunny" while legal tokenizer learns "jurisdiction", "plaintiff".

It runs three steps: (1) downloads data shards from HuggingFace, (2) trains BPE tokenizer on the data, (3) saves everything to ~/.cache/autoresearch/. Output: Data at ~/.cache/autoresearch/data/, Tokenizer at ~/.cache/autoresearch/tokenizer/. prepare.py is now complete! Next step: train.py.

18) **Step 9: train.py — Training Loop** — The evaluate() function runs the model on the validation set with dropout OFF (model.eval()) and returns val_bpb (bits per byte) — a vocab-size independent metric where lower is better, allowing fair comparison across different experiments. Understanding the distinction between tokenizer training and model training is critical to comprehending the system's architecture. These are two completely separate processes serving different purposes. Tokenizer training (prepare.py) learns how to split text into tokens, producing a dictionary/merge table that maps text sequences like "hello" to token sequences like [hel, lo]. This process takes 5 minutes per experiment, runs 100+ times with different hyperparameter configurations, produces neural network weights that evolve and improve, and is actively modified by the AI agent to search for optimal configurations. The analogy clarifies the distinction: tokenizer training creates an alphabet or dictionary that defines the vocabulary and how text maps to tokens, while model training learns to write coherent stories using that predefined alphabet. One defines the language, the other learns to use it effectively.

The GPT model architecture in train.py consists of four main classes implementing the Transformer architecture with careful attention to best practices. CausalSelfAttention implements multi-head attention where each token can only attend to previous tokens (not future ones), using Query, Key, Value projections plus a causal mask to enforce strict left-to-right information flow and prevent the model from "cheating" by seeing future tokens during training. FeedForward is an MLP (multi-layer perceptron) that runs after attention, consisting of Linear → GELU → Linear layers, expanding to 4x the embedding dimension then projecting back to the original size, allowing each token to independently process the information gathered by attention. TransformerBlock combines one attention layer and one feedforward layer with residual connections and LayerNorm in pre-norm style (LayerNorm before attention/FFN, which has been shown to be more stable than post-norm during training). The GPT class ties everything together into a complete language model: token embeddings convert token IDs to vectors, position embeddings encode sequence positions, N stacked TransformerBlocks (6 by default) progressively build representations, LayerNorm normalizes before the output layer, and a Linear layer predicts the next token. The architecture uses weight tying between input and output embeddings for parameter efficiency and includes a .generate() method for autoregressive text generation that can be used to evaluate model quality qualitatively. Key architectural concepts that make this work include the causal mask preventing future token visibility (essential for proper language modeling), residual connections (x + layer(x)) for stable gradient flow during training (solving the vanishing gradient problem in deep networks), and pre-norm architecture for improved training stability compared to the original Transformer's post-norm design.

A deep dive into the GPT architecture reveals the precise mechanics of each component and how they work together. CausalSelfAttention allows each token to look at previous tokens only (not future tokens) to decide what comes next, implementing the core "language modeling" constraint. The attention mechanism uses three learned transformations with clear semantic roles: Q (Query) asks "What am I looking for?", K (Key) answers "What do I contain?", and V (Value) provides "Here's my actual info". The information flow is: Q asks → K matches → V provides content, where attention scores determine which previous tokens are most relevant for predicting the next token. The `__init__` method sets up the tools (weights/layers) once at initialization — qkv_proj creates all three projections efficiently in a single matrix, out_proj blends attention heads, and the causal mask is registered as a buffer — while `forward` uses those tools on every batch during training. The output projection blends all attention heads into one unified output through learned mixing, allowing the model to combine information from multiple attention patterns. The causal mask is a lower triangular matrix that blocks future tokens (1 = see, 0 = blocked), enforcing "no peeking ahead" by setting future positions to negative infinity before softmax. Softmax converts raw attention scores to probabilities (values between 0-1 that sum to 1), creating a proper probability distribution over which tokens to attend to. Dropout randomly zeroes attention weights during training as a regularization technique, preventing the model from memorizing training data and forcing it to generalize by learning robust patterns rather than memorizing specific examples. FeedForward is a simple MLP that processes each token independently after attention has allowed tokens to exchange information. Its structure is: Linear (expand 4x the embedding dimension) → GELU (non-linear activation that filters what matters) → Linear (shrink back to original dimension) → Dropout (regularization). The key distinction in the architecture is: Attention allows tokens to talk to each other and exchange information based on learned patterns, while FeedForward lets each token digest and process that information alone without considering other tokens. The weights for both attention and feedforward are learned during training through backpropagation, not manually designed. TransformerBlock represents one complete layer combining both mechanisms: LayerNorm → Attention → Add (residual connection) → LayerNorm → FeedForward → Add (residual connection). Residual connections add the original input back to the layer's output, preserving information and enabling gradient flow in deep networks. LayerNorm stabilizes values by normalizing across the embedding dimension, preventing activation distributions from shifting during training. GPT stacks 6 blocks in sequence, and each round builds deeper understanding automatically — what each layer learns (low-level syntax in early layers, high-level semantics in late layers) emerges from training and is not manually defined by the programmer. The complete GPT Class implements the full pipeline: Token Embedding (convert token IDs to vectors) + Position Embedding (encode sequential positions) → 6 TransformerBlocks (progressively build representations) → LayerNorm (final normalization) → Linear head (project to vocabulary size for next token prediction). The `forward` method runs input through the model and returns both predictions and loss for training, while `generate` performs autoregressive text generation in a predict → append → repeat loop that can continue indefinitely to generate arbitrarily long sequences.

The tokenizer pipeline demonstrates memory-efficient data processing critical for handling large datasets on consumer hardware. list_parquet_files() scans DATA_DIR and returns a sorted list of .parquet files for deterministic processing order, while text_iterator() uses Python's yield keyword to create a generator that yields stories one-by-one from parquet files — a critical optimization that keeps only 1 story in RAM at a time instead of loading all 2.7M stories simultaneously, and stops after 100M characters (sufficient for tokenizer training without wasting time). The BPE (Byte Pair Encoding) tokenizer learns which character pairs to merge based on their frequency in the training dataset. The process starts with individual characters: "hello" begins as h e l l o, then the algorithm identifies and merges the most common character pairs iteratively to form subword units like hel lo. This approach has important properties: common words become single tokens (efficient), rare words split into multiple tokens (still representable), and the tokenizer can handle any text including previously unseen words (no "unknown token" problem). Training is inherently domain-specific: feeding your dataset text to the tokenizer allows it to count character pair frequencies across the entire corpus, converting the most frequent pairs into dedicated tokens in a way that optimizes for your specific domain — a TinyStories tokenizer learns tokens for "princess" and "bunny" because they appear frequently in children's stories, while a legal document tokenizer would learn specialized tokens for "jurisdiction" and "plaintiff" because those words dominate legal text. The autoresearch agent automates the tedious and expensive process of hyperparameter and architecture search, allowing researchers and practitioners to focus their effort on data selection and curation — the true differentiator for domain-specific models. Step 8 introduces the DataLoader class, which handles the critical task of feeding training data to the model in efficient batches for GPU processing. The DataLoader follows a four-step pipeline: reading parquet files from disk into memory, tokenizing text into token IDs using the trained tokenizer from prepare.py, filling an internal buffer with tokens for efficient sequential access, and creating batches of (input, target) pairs formatted as tensors ready for GPU processing. The input/target relationship is fundamental to language modeling and worth understanding clearly: inputs and targets are identical sequences offset by exactly one token position to create the "next token prediction" task. For example, if the input sequence is [Once, upon, a, time, the], the corresponding target sequence is [upon, a, time, the, princess] — shifted by exactly one position so the model learns to predict the next token given all previous tokens. The DataLoader implements several key methods that work together: `_load_next_file()` loads parquet files into memory one at a time for memory efficiency, `_get_next_tokens()` tokenizes one document at a time and returns token IDs, `_fill_buffer()` maintains a full token buffer for smooth batch generation without stalls, and `get_batch()` returns properly formatted (inputs, targets) tensors with correct shapes and device placement ready for immediate GPU consumption. The data split strategy is straightforward and effective: training uses all parquet files except the last one for maximum training data, while validation uses only the last file, ensuring clean separation between training and validation data with zero overlap to provide unbiased performance estimates.

Step 9 completes train.py with the training loop implementation that ties all components together into a working system. The evaluate() function runs the model on the validation set with dropout disabled via model.eval() to get deterministic predictions, and returns val_bpb (validation bits per byte) — a vocabulary-size independent metric where lower values indicate better performance, enabling fair comparison across different experiments that may use different hyperparameters or even different vocabulary sizes. The train() function implements the main training loop with a clear five-step process that runs for exactly TIME_BUDGET (5 minutes). The steps are: load the tokenizer from disk using the artifact created by prepare.py, create the GPT model instance with current hyperparameters (which may have been modified by the AI agent), create the AdamW optimizer with current learning rate and weight decay settings, execute the training loop for TIME_BUDGET (5 minutes) where each iteration gets a batch of (inputs, targets) from the DataLoader, performs a forward pass where loss = model(inputs, targets) calculates prediction error via cross-entropy loss, performs a backward pass where loss.backward() computes gradients via backpropagation showing how each parameter should change, and updates weights via optimizer.step() that adjusts parameters in the direction that reduces loss, and finally evaluate the trained model on the validation set and report val_bpb as the final metric for this experiment. The training loop implements several critical concepts that are often misunderstood: optimizer.zero_grad() clears old gradients before each step to prevent accumulation from previous batches (gradients add by default in PyTorch, so they must be explicitly cleared), loss.backward() performs backpropagation to compute partial derivatives showing how each weight should be adjusted to reduce loss, optimizer.step() actually updates the model weights based on the computed gradients and the optimizer's update rule (Adam in this case), and model.train() / model.eval() toggle dropout and other training-specific behaviors on or off for training versus evaluation modes. The entry point `if __name__ == "__main__": train()` makes train.py directly executable as a standalone script, allowing both the AI agent and human developers to run training experiments simply by executing the file with `python train.py` without requiring any wrapper scripts.

Step 10 introduces program.md, the instruction file that tells the AI agent (Claude or Codex) exactly what actions to perform during overnight autonomous experiments. The AI agent executes a continuous loop consisting of five distinct steps that repeat until stopped: modify train.py by adjusting hyperparameters (like learning rate, batch size) or architecture (like number of layers, attention patterns), run `python train.py` to execute a fixed 5-minute training session with the modified code, check the resulting val_bpb metric reported at the end of training, keep the modification if val_bpb improved (lower is better) and save the modified train.py as the new baseline or discard the modification if val_bpb worsened and revert train.py to the previous best version, and repeat the loop with a new experiment. The AI agent has explicit permission to modify specific aspects of train.py: hyperparameters including N_LAYERS (model depth), N_HEADS (attention parallelism), N_EMBED (representation size), DROPOUT (regularization strength), BATCH_SIZE (training batch size), LEARNING_RATE (optimization step size), and WEIGHT_DECAY (L2 regularization); architecture choices like activation functions (GELU, ReLU, SiLU), position embeddings (learned, sinusoidal, RoPE), and attention patterns (full attention, sparse attention, local windows); and training configurations such as optimizers (AdamW, SGD, Lion), learning rate schedulers (cosine, linear warmup), gradient clipping (prevent exploding gradients), and mixed precision settings (fp16, bf16 for speed). However, the AI agent is explicitly restricted from modifying two critical components: prepare.py (which handles data download and tokenizer training and must remain stable) and TIME_BUDGET (which must remain fixed at 5 minutes to ensure fair comparison between experiments — varying training time would make val_bpb comparisons meaningless). Each experiment is logged in a structured format to experiments.jsonl, recording the specific change made (e.g., "increased LEARNING_RATE from 3e-4 to 5e-4"), the resulting val_bpb value as a quantitative metric, whether performance improved or worsened compared to the previous best baseline, and the decision to keep or discard the change based on the result. With program.md complete, all three core files are now finished and working together: prepare.py handles one-time setup creating the dataset and tokenizer artifacts, train.py contains the model and training code that the AI modifies and executes hundreds of times, and program.md provides the instructions that guide the AI agent's autonomous experimentation process in a clear, unambiguous way.

A critical practical consideration emerged during testing on Apple Silicon hardware: memory limitations required adjusting the default BATCH_SIZE. BATCH_SIZE=32 caused "MPS backend out of memory" errors on Apple M4 MacBook Air, even though M4 is a modern and capable chip. The root cause is that the self-attention mechanism has O(seq_len²) memory complexity — it must compute and store attention scores between every pair of tokens in the sequence. With seq_len=2048 (the maximum sequence length), this creates a 2048 × 2048 attention matrix per head, and with 6 heads per layer and 6 layers, memory usage quickly exceeds available GPU memory at batch size 32. The solution was straightforward: reduce BATCH_SIZE to 4, which fits comfortably in available memory. This change has several implications for training dynamics that are important to understand. Steps per 5 minutes increases by approximately 8x because the same amount of data is now processed in smaller chunks (8 times as many batches of size 4 to equal the same data as 1 batch of size 32). Gradient noise becomes noisier because each gradient update averages over fewer samples (4 instead of 32), making individual steps less accurate estimates of the true gradient direction. Convergence becomes slightly slower per step because noisy gradients take less optimal paths toward the minimum, but the much larger number of total steps (8x more) compensates for this by allowing more frequent weight updates. Final quality after sufficient training is nearly the same because the model will still learn the same patterns and relationships given enough steps — batch size is primarily a memory/speed tradeoff, not a fundamental capacity limitation. The conclusion is that batch size 4 is perfectly fine for this use case. The model trains normally and improves normally during autonomous optimization. Batch size mainly affects memory usage and training dynamics, not the final model quality after sufficient training time. Importantly, the AI agent still finds the same optimal hyperparameters (learning rate, number of layers, etc.) — the batch size reduction doesn't prevent the agent from discovering good configurations, it just changes the speed at which each individual experiment runs. This makes the autoresearch approach accessible even on consumer hardware with limited memory, democratizing autonomous LLM research beyond high-end workstations.
